# Project Knowledge — d74c6e5

## 入口与 CLI (bin/)

**文件**: `bin/cc.js`

CLI 入口，使用 Commander.js 构建命令行程序。程序名 `zcc`。

### 命令注册

所有命令均通过 `program.command()` 注册，每个命令委托给 `src/commands/` 下对应模块的导出函数。全局选项 `-t, --target <file>` 通过 `program.opts().target` 传递给各子命令。

### 关键设计细节

- **默认动作**（`zcc <profile-id>`）: 使用 `program.argument()` + `passThroughOptions()` 注册。`passThroughOptions()` 使得未知参数（如 `-c`、`--verbose`）不被 Commander 消费，而是通过 `command.args.slice(1)` 作为 `extraArgs` 透传给 `launchCommand`，最终传递给 `claude` CLI。
- **选项继承**: 每个子命令通过 `{ ...options, target: options.target || program.opts().target }` 合并全局选项。这导致子命令自己的 `-t` 优先于全局 `-t`。
- **serve 子命令嵌套**: `serve` 拥有子命令 `list`、`stop`、`log`，通过 `serveCmd.command()` 注册。`serveCommand(action, id, options)` 函数通过 `action` 参数区分操作类型——当 `action` 不是 `list`/`stop`/`log` 时，被视为 profile-id 走启动代理逻辑。
- **knowledge 子命令**: 接收 `<subcommand>` 参数，直接传递给 `knowledgeCommand`，不再嵌套注册。

### 边界情况

- 无参数调用 `zcc` 会显示 help（`program.help()`），而非报错。
- `serve` 的 `-t` 选项在 serveCmd 和其子命令 `stop`、`log` 上都重复声明，因为 Commander 不会从父命令继承选项。

## 配置层 (src/config/)

包含 5 个文件：`loader.js`、`validator.js`、merger.js`、`env-registry.js`、`env-selector-prompt.js`。

### loader.js — 配置读写与 profile 解析

**核心常量**:
- `GLOBAL_CONFIG_PATH`: `~/.claude/models.yaml`
- `LOCAL_CONFIG_PATH`: `.claude/models.yaml`（相对路径，运行时 resolve）

**关键函数**:

- `loadConfig(customConfigPath?)`: 读取 YAML 配置。无参数时读全局。若目标文件不存在，自动创建 `~/.claude/` 目录并写入默认配置（`{ settings: { alias: 'cc' }, base: {}, profiles: {} }`），返回默认值的浅拷贝。
- `saveConfig(config, customConfigPath?)`: 写入 YAML，自动创建目录。无路径时写全局。
- `loadGlobalConfig()`: 仅在文件存在时加载，否则返回 `null`。不自动创建。
- `findProfile(profileId, customConfigPath?, options?)`: **三级优先级查找** —— custom > local > global。始终预加载全局配置用于 base 继承。返回 `{ profile, configPath, source }`。`options.mergeBase` 默认 `true`。
- `resolveProfile(globalConfig, localConfig, profileId)`: 继承链 `deepMerge(deepMerge(globalBase, localBase), profile)`。
- `getSettingsDir(configPath)`: 返回配置文件所在目录，用于生成 `settings.<id>.json` 路径。

**非显而易见的细节**:
- `loadConfig` 对不存在的自定义路径也会创建默认文件（因为 `targetPath` 解析后走同一逻辑），这可能导致意外的文件创建。
- `findProfile` 在 global 级查找时调用 `resolveProfile(null, globalConfig, profileId)`，传入 `null` 作为 globalConfig，这意味着全局 base 仍然通过 `localConfig.base` 路径生效。
- `DEFAULT_CONFIG` 中 `alias` 是 `'cc'`，但 `bin/cc.js` 中 `program.name('zcc')`。

### validator.js — ID 校验

- `validateConfigId(id, existingConfigs)`: 正则 `/^[a-zA-Z0-9._-]+$/`，同时检查是否已存在。返回 `{ valid, error }`。
- `validateConfigEntry(entry)`: 仅检查是否为非空对象。

### merger.js — 深度合并

- `deepMerge(base, override)`: 递归合并。数组行为是 **拼接去重**（`[...new Set([...base, ...override])]`）。标量值 override 覆盖 base。`undefined` 时保留另一侧。

**边界情况**:
- 对 Set 的去重依赖 JSON 序列化等价性（数组元素为原始值时正确，对象引用不保证）。
- `deepMerge` 是不可变操作（不修改输入），返回新对象/数组。

### env-registry.js — 环境变量注册表

**约 200 个内置环境变量**，按 12 个分类（Provider, Auth, Model, Network, MCP, Privacy, Context, Shell, Feature, Plugin, OTelemetry, Vertex, Display）组织。每个条目有 `{ key, category, desc, type }`，type 为 `flag`/`text`/`number`/`choice`。

**关键函数**:

- `loadEnvRegistry()`: 三层合并 —— `BUILTIN_ENV_VARS` + `~/.claude/env-registry.yaml` + `./.claude/env-registry.yaml`，后者覆盖前者。通过 `mergeEntries()` 实现按键匹配的覆盖。
- `buildPagedEnvSource(entries, existing)`: 返回 `{ source, controller }`。`source` 是 inquirer autocomplete 的 source 函数。`controller` 支持左右箭头切换分类，带循环索引。搜索输入时跨分类搜索，无输入时显示当前分类。
- `buildAutocompleteSource(entries, existing)`: 旧版不分页的 source 函数（按分类分组显示全部）。
- `promptEnvValue(varDef, currentValue)`: 根据 type 选择不同的 inquirer prompt 类型。flag 返回 `'1'` 或 `null`（不是布尔值）。
- `appendToRegistry(entry, scope)`: 追加到 YAML 文件。不重复添加。

**非显而易见的细节**:
- `saveEnvRegistry` 只保存非内置条目，防止内置变量被写入用户文件。
- `buildPagedEnvSource` 的搜索模式下，已设置的环境变量（`existing` 中的键）被过滤掉，不在候选列表显示。
- `promptEnvValue` 对 flag 类型返回 `'1'`（字符串）而非 `true`，因为环境变量需要字符串值。

### env-selector-prompt.js — 自定义 inquirer prompt

继承 `inquirer-autocomplete-prompt`，扩展左右箭头键切换分类功能。通过 `opt.sourceController` 注入控制器。按左右箭头时清空搜索输入（`this.rl.line = ''`），重置选中项到第 0 个。

## 命令层 (src/commands/)

12 个命令文件。每个导出一个主函数，接收 Commander 解析的参数和选项。

### launch.js — 启动 Claude Code

`launchCommand(profileId, options, extraArgs)`

**流程**:
1. `findProfile(id, target, { mergeBase: false })` — 不合并 base
2. 检测 `profile.proxy` 字段，调用 `checkProxyAlive()` 做健康检查
3. 若代理存活：生成 `settings.<id>.proxy.json`，将 `ANTHROPIC_BASE_URL` 替换为代理地址，删除 `proxy` 字段
4. 若代理已停止：从配置中清除 `proxy` 字段，降级为直连模式
5. 写入 settings JSON 文件，`spawn('claude', ['--settings', path, ...extraArgs])`

**非显而易见的细节**:
- Windows 平台使用 `shell: true` 模式 spawn，将整个命令拼接为字符串执行；其他平台使用参数数组。这是因为 Windows 上 `claude` 可能是 `.cmd` 文件。
- `mergeBase: false` 是关键设计——避免覆盖 Claude Code 自动加载的全局 `~/.claude/settings.json`。
- 代理降级时，先 `loadConfig` 再删除 proxy 再 `saveConfig`，确保不丢失其他并发修改。

### list.js — 列出 profiles

`listCommand(options)`

只读操作，显示配置文件路径和每个 profile 的 env 键及 settings 键摘要。本地配置不存在时显示空列表（不创建默认文件）。

### add.js — 添加 profile

`addCommand(profileId, options)` — 异步函数（inquirer 交互）

**两种模式**:
- `--base` 模式: 编辑 `config.base`，最终使用 `deepMerge(existingBase, base)` 合并
- 普通模式: 新建 profile，验证 ID 唯一性后直接赋值

**`-s, --source <file>`**: 从 settings JSON 预填充值。核心变量（BASE_URL, AUTH_TOKEN, MODEL, SUBAGENT_MODEL）由交互提示处理，非核心变量自动预填。

**环境变量选择循环**: 使用 `env-selector` prompt，支持左右箭头分类切换。选择 `__custom__` 进入手动输入。输入后调用 `maybeSaveToRegistry` 提示保存到注册表。

**导出的辅助函数**:
- `maybeSaveToRegistry(key, registry)`: 非 builtin 且非已注册时，询问用户是否保存
- `promptSubagentModel(modelValue, existingValue)`: 仅当设置了 ANTHROPIC_MODEL 时才提示，先问是否一致，再允许手动输入

### edit.js — 编辑 profile

`editCommand(profileId, options)` — 复用 add.js 的 `maybeSaveToRegistry` 和 `promptSubagentModel`

与 add 的关键差异:
- 允许修改 profile ID（重命名）
- 使用 `manageExistingEnvVars()` 显示 checkbox 删除非核心变量
- 保存时直接赋值（非 deepMerge），会覆盖整个 profile
- base 模式使用 `deepMerge(existingBase, base)` 合并

### remove.js — 删除 profile

`removeCommand(profileId, options)` — 简单的 `delete config.profiles[id]`，本地配置不存在时报错。

### alias.js — 修改 CLI 别名

`aliasCommand(newAlias, options)` — 无参数时显示当前别名。总是操作全局配置或自定义路径。注意：修改 alias 只改 YAML，不影响实际 PATH 中的命令名。

### use.js — 应用 profile 到 settings 文件

`useCommand(profileId, options)`

**流程**:
1. 加载 profile（`--base` 模式直接取 `config.base`，否则 `findProfile` 默认合并 base）
2. 确定目标路径: `--global` 写 `~/.claude/settings.json`，否则写 `./.claude/settings.local.json`
3. **一次性备份**: 若 settings 文件存在且 `settings.source.json` 不存在，复制为 source
4. 合并: `deepMerge(sourceSettings, profile)` — source（用户原始配置）作为 base，profile 覆盖
5. 写入

**非显而易见的细节**:
- 备份只在 source 不存在时创建，永不覆盖。多次 `use` 不同 profile 时，始终以第一次 use 前的状态为 base。
- `--base` 模式下 profile 查找优先本地，不存在才回退全局。

### restore.js — 恢复 settings 备份

`restoreCommand(options)` — 将 `settings.source.json` 复制回 `settings.json` 或 `settings.local.json`。

### parse.js — 导入 settings JSON 为 profile

`parseCommand(settingsPath, profileId, options)`

**三种模式**:
- `--copy`: 格式化为 YAML 复制到剪贴板
- `--base`: `deepMerge(existingBase, settings)` 合并到 base
- 普通模式: profile-id 冲突时弹出交互选择（自动后缀 / 重命名 / 取消）

**辅助函数**:
- `copyToClipboard(text)`: 跨平台剪贴板（clip / pbcopy / xclip）
- `findAvailableSuffix(baseId, profiles)`: 生成 `<id>-1`, `<id>-2` 等不冲突名称

### serve.js — 本地代理管理

`serveCommand(action, id, options)` — 路由到子操作

**子操作**:
- 默认（无 action）→ `startProxy(profileId, options)`
- `list` → `listProxies()`
- `stop` → `stopProxy(profileId, options)` 或 `stopAllProxies()`
- `log` → `logProxy(profileId, options)`

**startProxy 流程**:
1. 查找 profile 或 base 配置
2. 验证 `env.ANTHROPIC_BASE_URL` 存在
3. 检查重复启动保护（通过 `checkProxyAlive`）
4. 生成随机 token，构建 proxyConfig
5. `fork(worker.js, [JSON.stringify(proxyConfig)])` 启动子进程，stderr 重定向到日志文件
6. 通过 IPC 等待 ready 消息（10 秒超时）
7. 写入 `proxy` 字段到配置（`{ url, pid, port, token }`）
8. `child.disconnect()` + `child.unref()` 使子进程独立运行
9. `--run` 模式: 调用 `launchCommand`

**scanProxies()**: 扫描本地和全局配置中所有含 `proxy` 字段的 profile 和 base。

**listProxies()**: 显示状态并自动清理已停止的代理（删除 proxy 字段）。

**非显而易见的细节**:
- worker 的 stderr 被重定向到日志文件，便于崩溃诊断。
- `startProxy` 使用 `var profile`（在 isBase 分支中），这是有意为之——在 else 分支中通过 `const` 声明的 `profile` 在外层也可访问，因为 `var` 是函数作用域。
- 代理停止时直接 `process.kill(pid)`，不等待优雅退出。
- `checkProxyAlive` 通过 HTTP GET `/__cc_proxy_status?token=xxx` 验证，3 秒超时。

### test.js — 测试 API 连接

`testCommand(profileId, options)` — 调用 `api/client.js` 的 `sendApiRequest` 请求 `/models` 端点。base 模式查找逻辑与 serve.js 一致（优先本地）。

### models.js — 查询可用模型

`modelsCommand(profileId, options)` — 调用 `sendApiRequest` 请求 `/models`，用 `extractModelNames` 解析响应。支持多种响应格式（`data.data`、`data.models`、顶级数组）。

### knowledge.js — 知识库管理

`knowledgeCommand(subcommand, options)` — 四个子命令

**核心数据结构**: `.knowledge/index.json` 记录各 section 的 base commit、当前知识文件名、上一个文件名。

**sections 定义**: `{ bin, config, commands, proxy, api }`，每个 section 对应一组源码路径。

**status**: 对每个 section 执行 `git diff --numstat`，通过 `classifyChange` 分为 `unchanged`/`minor`/`significant`。
- minor: <5 行变更且 <3 文件
- significant: >=5 行 或 >=3 文件 或 包含新增/删除文件

**update**: 收集 significant stale sections，生成 diff 摘要（截断 2000 字符），原子写入新知识文件，更新 index。若 >=2 个 stale section 则标记 `cross-module`；若 config stale 则标记 `non-obvious`。

**rebuild**: 从零创建 `.knowledge/` 目录、index.json 和带骨架的知识文件。

**verify**: 检查 index.json 完整性、文件存在性、commit 有效性、章节标题匹配。

**辅助工具**:
- `atomicWrite()`: 先写临时文件再 rename
- `generateFileName()`: `YYYY-MM-DD-<short-hash>.md` 格式，冲突时追加序号
- `cleanupTempFiles()`: 清理 `.tmp-` 前缀的临时文件

## 代理层 (src/proxy/)

### server.js — HTTP 反向代理

`createProxyServer(options)` → `http.Server`

**请求处理流程**:
1. 健康检查: `GET /__cc_proxy_status?token=xxx` → 返回 `{ status: 'ok', pid }`
2. 收集请求体（`for await (chunk of req)`）
3. JSON 解析 → 模型替换逻辑:
   - `modelOverride[source]` 映射 → 替换
   - 无映射但有 `defaultModel` → 强制替换
   - 记录 `originalModel` 和 `replacedModel`
4. 构建上游请求: path 拼接 base pathname + request path
5. 响应透传（`upstreamRes.pipe(res)`）
6. 日志记录（通过 `logger.push()`）

**非显而易见的细节**:
- HTTPS 上游设置 `rejectUnauthorized: false`，支持自签证书
- 非 JSON 请求体透传不修改
- 请求体中无 `model` 字段时不做替换
- `content-length` 在模型替换后重新计算（用新 body 的 `Buffer.length`）
- 删除了 `connection` 和 `transfer-encoding` 头防止代理问题
- `pushLog` 有防重入保护（`logged` 标志），确保每个请求只记录一次

### worker.js — 后台进程入口

接收 `process.argv[2]` 的 JSON 配置。创建 logger 和 server。`server.listen(0)` 由操作系统分配随机端口。通过 `process.send({ type: 'ready', port, pid })` IPC 通知父进程。

**信号处理**:
- `disconnect`: 不退出（保持进程存活）
- `SIGTERM`: `flushSync()` 后退出（优雅关闭）
- `uncaughtException`: IPC 发送错误消息 + `flushSync()` 后退出

### logger.js — 异步日志队列

`createLogger(options)` → `{ push, flush, flushSync, stop, isDisabled }`

**核心机制**:
- 所有 `push()` 操作同步入队 + 唤醒 drain 循环（`setImmediate`）
- drain 循环批量写入（`fs.appendFile`），一次取出所有队列条目
- 队列溢出（>= 10000）时丢弃最旧的条目，记录丢弃计数
- 丢弃的条目在下次 batch 前插入 `{ type: 'dropped', count }` 标记
- 连续写入失败 >= 3 次自动禁用 logger
- `flushSync()` 使用 `appendFileSync`，专用于崩溃路径

**非显而易见的细节**:
- 日志文件路径懒初始化（首次写入时创建），目录不存在时自动创建
- `stop()` 内部用同步写入确保所有剩余条目落盘
- `push()` 在 `stopped` 或 `disabled` 状态下静默丢弃，不报错
- `scheduleDrain` 可能被递归调用（写入完成后队列又有新条目时），但通过 `draining` 标志防止并发

## API 层 (src/api/)

### client.js — HTTP 请求封装

`sendApiRequest(baseUrl, token, options)` → `Promise<{ statusCode, statusMessage, durationMs, body, data }>`

**行为**:
- 自动去除 baseUrl 尾部斜杠
- 同时设置 `Authorization: Bearer` 和 `x-api-key` 头（兼容不同 API 实现）
- HTTPS 时 `rejectUnauthorized: false`
- 默认 10 秒超时
- 响应体尝试 JSON 解析，失败时 `data` 为 `null`

**非显而易见的细节**:
- URL 解析失败时 reject 一个中文错误消息 `'无效的 URL: ...'`，`models.js` 依赖此消息做特定错误处理
- 此模块被 `test.js` 和 `models.js` 共用

### 关键交互模式

1. **配置读写模式**: 几乎所有命令都依赖 `loader.js`。`loadConfig` + 修改 + `saveConfig` 是标准读写模式。`findProfile` 是只读查询模式。
2. **profile 查找策略差异**: `launch` 使用 `{ mergeBase: false }`，`use`/`test`/`models` 使用默认 `true`。
3. **serve ↔ launch 循环**: `serve --run` 启动代理后调用 `launchCommand`；`launch` 启动时调用 `checkProxyAlive`（来自 serve.js）。
4. **add ↔ edit 共享**: `edit.js` 从 `add.js` 导入 `maybeSaveToRegistry` 和 `promptSubagentModel`。
5. **api/client 共享**: `test.js` 和 `models.js` 共用 `sendApiRequest`。
6. **proxy 层隔离**: `worker.js` 通过 `fork()` 作为独立进程运行，通过 `process.argv[2]` JSON 传参、IPC 消息通信。`server.js` 和 `logger.js` 不直接被任何命令导入。

## 非显而易见的细节

1. **配置加载副作用**: `loadConfig()` 对不存在的路径会创建文件并写入默认配置。仅 `loadGlobalConfig()` 是纯读取（不存在返回 null）。`list.js` 绕开了这个问题：先检查文件存在再调用 `loadConfig`。

2. **base 继承的不对称性**: `findProfile` 在 global 级找到 profile 时，调用 `resolveProfile(null, globalConfig, profileId)`。此时 `globalBase` 为 `null?.base = undefined`，`localBase` 为 `globalConfig.base`。效果是全局 profile 只继承全局 base，不继承本地 base。这是正确行为但实现不直观。

3. **use 的备份策略**: `settings.source.json` 只创建一次，永远不更新。这意味着第一次 `zcc use A` 后再用 `zcc use B`，B 合并的 base 仍然是原始 settings（不是 A 的结果）。

4. **Windows spawn 差异**: `launch.js` 在 Windows 上使用 `shell: true` + 字符串命令，其他平台使用参数数组。这会影响参数中的特殊字符处理。

5. **serve 的 proxy 字段生命周期**: proxy 字段由 `startProxy` 写入配置，由 `launch` 读取并可能删除（代理已停止时），由 `listProxies` 清理已停止条目，由 `stopProxy`/`stopAllProxies` 主动清理。多个命令都可以修改同一份配置文件。

6. **logger 的原子性**: `logger.push()` 是同步入队，但写入是异步的。崩溃时可能丢失最后几条日志，除非调用 `flushSync()`。worker.js 的 `SIGTERM` 和 `uncaughtException` 处理器都调用了 `flushSync()`。

7. **flag 环境变量的值**: `promptEnvValue` 对 flag 类型返回 `'1'`（启用）或 `null`（跳过）。不会返回 `'0'` 或 `false`。用户无法通过交互界面显式设置为 `"0"`。

8. **edit 的保存语义**: 普通 profile 编辑直接赋值（覆盖），但 base 编辑使用 `deepMerge(existingBase, base)` 合并。这意味着编辑普通 profile 时，未在交互中显示的字段（如 `hooks`、`modelOverride`）会通过 `Object.assign` 保留。

9. **parse 的冲突处理**: profile ID 冲突时使用异步 inquirer 交互（`handleConflict`），但无冲突路径是同步的（`doSave`）。`parseCommand` 函数本身是同步声明的，但可能返回 Promise。

10. **knowledge 的 git 依赖**: 整个 knowledge 模块依赖 `git` CLI（通过 `execSync`），在非 git 仓库中会完全不可用。`getHeadCommit` 失败会导致 rebuild 报错。

11. **serve 的子进程分离**: worker 进程使用 `detached: true` + `unref()` 运行。父进程退出后子进程继续运行。这意味着 orphaned worker 进程需要通过 `zcc serve stop` 或 PID kill 手动清理。

12. **modelOverride 的优先级**: 在 `server.js` 中，`modelOverride` 映射优先于 `defaultModel`。只有当请求的 model 不在 override 表中时，才会使用 defaultModel。两者都不存在时，请求体中的 model 保持不变。
