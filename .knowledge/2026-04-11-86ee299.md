## 入口与 CLI (bin/)

### 模块职责
`bin/cc.js` 是整个 CLI 应用程序的执行入口文件。它基于 `commander` 库构建命令行界面，定义了名为 `zcc` 的指令集。其主要职责是解析用户的命令行参数、选项，并将其路由分发到 `src/commands/` 目录下对应的业务逻辑处理模块中。

### 全局选项
- `-t, --target <file>`：指定自定义的配置文件路径（YAML 格式）。该选项可被多个子命令继承和使用。

### 核心命令与导出接口

该模块本身不直接导出接口，而是通过注册命令来触发其他模块的函数。它包含以下主要命令及对应的核心函数：

1. **默认主命令 (`zcc <profile-id>`)**
   - **行为**：如果未提供 `profile-id`，则自动输出帮助信息。支持透传额外的参数。
   - **依赖**：`launchCommand`（来自 `src/commands/launch`）。

2. **配置与增删改查**
   - `list`：列出所有配置。依赖 `listCommand`。
   - `add <profile-id>`：添加新配置，支持从 JSON 文件导入 (`-s, --source`) 或编辑基础配置 (`-b, --base`)。依赖 `addCommand`。
   - `remove <profile-id>`：移除指定的配置。依赖 `removeCommand`。
   - `edit <profile-id>`：编辑现有配置。依赖 `editCommand`。
   - `use [profile-id]`：将配置应用到 settings 文件但不启动服务。依赖 `useCommand`。

3. **工具与环境命令**
   - `alias [name]`：修改 CLI 的命令别名。依赖 `aliasCommand`。
   - `restore`：从备份文件 (`settings.source.json`) 恢复设置。依赖 `restoreCommand`。
   - `parse <settings-path> [profile-id]`：将外部的 settings JSON 文件解析并保存为 YAML 配置文件，支持复制到剪贴板 (`-c, --copy`)。依赖 `parseCommand`。
   - `test [profile-id]`：测试指定配置的 API 连通性。依赖 `testCommand`。
   - `models [profile-id]`：查询指定配置可用的模型列表。依赖 `modelsCommand`。

4. **高级功能子系统**
   - `knowledge <subcommand>`：用于管理项目知识库（提供 `status/update/verify/rebuild` 等子命令），支持指定更新板块 (`--section`) 和使用 AI 分析的配置 (`--profile`)。依赖 `knowledgeCommand`。
   - `serve`：管理本地模型代理服务器。依赖 `serveCommand`。
     - `serve [profile-id]`：启动代理，支持直接运行 (`--run`)。
     - `serve list`：列出正在运行的代理服务。
     - `serve stop [profile-id]`：停止特定的代理服务或全部代理 (`--all`)。
     - `serve log <profile-id>`：查看代理的请求日志，支持限制显示行数 (`-n, --lines`)。

### 模块依赖关系
- **核心依赖**：`commander`（用于构建命令行骨架）。
- **命令分发依赖**：该文件不包含业务逻辑，而是深度依赖 `../src/commands/` 目录下的独立模块。所有命令行解析和参数组装完成后，统一调用对应模块暴露的 Command 函数（如 `launchCommand`, `listCommand`, `knowledgeCommand` 等），从而实现入口与业务逻辑的完全解耦。
## 配置层

配置层负责管理整个 CLI 工具的多层级配置体系，涵盖配置文件的深度合并、环境变量的注册与交互式管理，以及配置参数的校验。它为应用运行提供了灵活且结构化的环境参数设置能力。

### 模块依赖关系
- `loader.js` 依赖 `merger.js` 提供的 `deepMerge` 能力来实现多层级配置的合并。
- `env-selector-prompt.js` 依赖社区库 `inquirer-autocomplete-prompt` 来扩展实现带有分类切换功能的交互式选择器。
- `env-registry.js` 依赖 `inquirer` 和 `js-yaml` 来实现环境变量的交互式收集与 YAML 文件的读写。

### 核心模块

#### 1. 配置加载器
配置加载器维护全局 (`~/.claude/models.yaml`) 和本地 (`.claude/models.yaml`) 的双层配置体系。

- **`loadConfig(customConfigPath)`**: 加载并解析配置。若未指定路径且全局配置不存在，将自动初始化并写入默认配置（包含 `settings`、`base` 和 `profiles` 块）。
- **`findProfile(profileId, customConfigPath, options)`**: 查找并解析特定的配置档案。查找优先级为：自定义路径 > 本地配置 > 全局配置。
- **`resolveProfile(globalConfig, localConfig, profileId)`**: 解析并合并配置档案。采用级联基础继承机制，合并优先级为：`globalBase` → `localBase` → `profile`，上层配置覆盖底层。
- **`getSettingsDir(configPath)`**: 根据传入的 `configPath` 推断出对应的 `settings.<id>.json` 文件应写入的目录。

#### 2. 合并器
提供符合特定合并语义的深度合并函数。

- **`deepMerge(base, override)`**: 遵循以下规则合并两个值：
  - **数组**：连接两个数组并去除重复项。
  - **对象**：递归合并共有的键。
  - **标量**：直接使用 `override` 的值覆盖。

#### 3. 环境变量注册表
环境变量注册表管理着超过 150 种内置环境变量，并支持在全局或本地作用域内的自定义扩展。

- **变量分类**: 包含 Provider (服务商)、Auth (认证)、Model (模型)、Network (网络)、MCP、Privacy (隐私)、Context (上下文)、Shell、Feature (特性)、Plugin (插件)、OTelemetry (遥测)、Vertex、Display (显示) 等多个分类。
- **`loadEnvRegistry()`**: 加载环境变量定义。按 内置定义 → 全局覆盖 (`~/.claude/env-registry.yaml`) → 本地覆盖 (`.claude/env-registry.yaml`) 的顺序加载与合并。
- **`saveEnvRegistry(entries, scope)` / `appendToRegistry(entry, scope)`**: 将用户自定义的环境变量持久化保存到指定作用域（local 或 global）的 YAML 文件中。
- **交互式 UI 构建**:
  - `buildEnvChoices`: 构建分类别、带分隔符的列表选项。
  - `buildAutocompleteSource`: 构建支持按 Key 或描述模糊搜索的自动补全数据源。
  - `buildPagedEnvSource`: 构建分类分页数据源，返回包含 `source` 函数与 `controller` 控制器的对象，用于左右键翻页。
- **`promptEnvValue(varDef, currentValue)`**: 根据变量类型（flag、choice、number、text）动态生成对应的交互式提示 UI 以收集输入值。

#### 4. 自定义交互提示
继承自 `AutocompletePrompt` 的自定义交互式提示组件，专为环境变量选择场景设计。

- **分类切换**: 拦截键盘的 `Left` 和 `Right` 按键事件，调用 `sourceController.switchCategory` 实现上一分类 / 下一分类的切换，切换时会自动清空搜索词并重置选中状态。
- **按键代理**: 其他常规按键事件（如上下选择、输入、回车）均代理给父类处理。

#### 5. 配置校验器
提供针对配置项的基础校验能力。

- **`validateConfigEntry(entry)`**: 校验传入的配置项是否为非空的对象结构。
- **`validateConfigId(configId, existingConfigs)`**: 校验配置 ID 是否合法（仅允许字母、数字、点、连字符和下划线），并确保其不会与已有的配置项发生命名冲突。

- **validator.js**: `{ validateConfigEntry, validateConfigId }`

## 命令层

命令层 (`src/commands/`) 是 CLI 工具的核心交互入口，负责解析用户输入、调度配置层与代理层，并处理交互式提示。它通过 `inquirer` 提供丰富的终端交互体验。

### 模块职责与导出接口

- **add**: 通过交互式提示引导用户创建新的配置档案或全局基础配置。支持从外部 JSON 文件导入配置（`--source`），并能在设置环境变量时调起自定义的分页选择器。
- **edit**: 修改已有的配置档案。具备 `add` 的所有交互功能，额外支持重命名档案 ID 以及通过勾选框删除已有的环境变量。
- **remove**: 从本地或全局的 `models.yaml` 配置文件中移除指定的配置档案。
- **list**: 扫描并列出配置文件中所有可用的配置档案，显示其包含的环境变量及其他设置项。
- **alias**: 查看、修改 CLI 工具的命令别名（默认为 `zcc`）。
- **parse**: 将外部的 Claude Code `settings.json` 文件解析并导入到本工具的配置体系中。支持转化为 YAML 并复制到剪贴板，或在遇到同名档案时提供重命名/覆盖的交互提示。
- **use**: 将指定的配置档案应用到当前目录或全局目录的 Claude Code 环境中。它会自动备份原始的 `settings.json`（命名为 `settings.source.json`），并利用深度合并将用户原设置与新档案结合。
- **restore**: 利用 `use` 命令生成的备份文件，一键恢复原始的 Claude Code 环境设置。
- **models**: 根据指定档案内配置的 API 凭证，向 `/models` 端点发起请求，查询并列出当前可用的模型列表。
- **test**: 验证指定档案的 API 连通性。通过请求 `/models` 端点来快速反馈连接是否成功、认证是否失败或服务器不可达。
- **launch**: 查找指定档案，将其配置生成为一个临时的 `settings.<id>.json` 文件，然后作为子进程直接启动 Claude Code 并应用该设置。
- **serve**: 代理服务管理器。包含启动、停止、列出和查看日志四个子命令。在启动代理时，它会通过 `fork` 创建一个 detached 的代理子进程，生成鉴权 token，监听就绪消息，并将代理状态写回到配置文件中。
- **knowledge**: 项目知识库管理命令。基于 Git commit hash 追踪代码变更与知识库的时效性。包含四个子命令：
  - `status`: 对比当前 HEAD 与索引中记录的 commit，使用 `git diff --numstat` 评估变更规模，将知识库各章节状态标记为 up to date、minor 或 stale。
  - `update`: 增量更新。针对 stale 章节获取具体 diff，结合提供的 profile 调用 AI 接口重写对应章节。
  - `verify`: 校验知识库结构的完整性，包括文件存在性、commit 有效性和章节标题匹配度。
  - `rebuild`: 从零开始重建知识库。读取各命令层源码，调用 AI 生成全新的项目知识文档。

### 核心函数

- `promptSubagentModel` (add): 在设置 `ANTHROPIC_MODEL` 后触发，智能提示用户是否将 `CLAUDE_CODE_SUBAGENT_MODEL` 保持一致或单独配置。
- `maybeSaveToRegistry` (add): 当用户在交互中手动输入了非内置的环境变量时，提示将其保存到本地或全局的环境变量注册表（`env-registry.yaml`），方便日后复用。
- `extractSection / replaceSection` (knowledge): 从完整的 Markdown 文档中查找指定的二级标题块，提取或替换其内容，用于实现知识库的局部热更新。
- `checkProxyAlive` (serve): 通过 HTTP 请求代理特有的 `/__cc_proxy_status` 端点并验证 token 参数，确保代理进程真实存活。
- `scanProxies` (serve): 扫描本地与全局的所有配置文件，收集所有包含 `proxy` 字段的配置档案和基础配置。

### 模块依赖关系

- **配置读取**: 所有命令均高度依赖 `../config/loader`（如 `loadConfig`, `findProfile`, `getLocalConfigPath` 等）来定位和解析 YAML 配置文件。
- **数据校验与合并**: `add`、`edit`、`parse`、`use` 依赖 `../config/validator` 进行 ID 格式校验，依赖 `../config/merger` 进行配置项的深度合并。
- **网络请求**: `models`、`test`、`knowledge` 依赖 `../api/client` 的 `sendApiRequest` 函数来与外部 LLM API 交互。
- **环境变量注册**: `add` 和 `edit` 依赖 `../config/env-registry` 及自定义的 `../config/env-selector-prompt` 来提供可搜索、可分页的环境变量选择界面。
- **代理层交互**: `serve` 通过 `fork` 启动 `../proxy/worker.js`，而 `launch` 则通过检测 `serve` 写入的 `proxy` 字段来判断是否将请求重定向到本地代理。
- **命令间复用**: `edit` 直接从 `add` 导入 `maybeSaveToRegistry` 和 `promptSubagentModel`；`launch` 被 `serve` 在使用 `--run` 参数时动态引入。
## 代理层 (src/proxy/)

代理层负责在本地启动一个轻量级的 HTTP 反向代理服务，主要用途是将客户端的 API 请求转发给上游服务器，同时支持在转发前动态修改请求内容（如模型替换），并对所有经过代理的请求进行异步日志记录。该层被设计为一个完全独立的后台进程。

### 模块职责与核心函数

#### 1. 入口与进程管理 (`worker.js`)
作为独立后台代理进程的启动入口，负责进程的初始化、生命周期管理及通信。
- **初始化配置**：解析来自父进程通过命令行参数 (`process.argv[2]`) 传入的 JSON 配置。
- **组件拼装**：根据配置实例化日志写入器 (`createLogger`) 和代理服务器 (`createProxyServer`)。
- **动态端口监听**：调用 `server.listen(0)` 让操作系统自动分配可用端口。
- **IPC 通信**：服务启动后，通过 `process.send` 向父进程发送包含分配端口和进程 PID 的 `ready` 消息。
- **生命周期与信号处理**：
  - `SIGTERM`：响应停止信号，触发日志的同步刷写后安全退出。
  - `uncaughtException`：捕获未知异常，通过 IPC 发送 `error` 消息给父进程，同步刷写日志并退出，防止进程静默崩溃。

#### 2. 代理服务器 (`server.js`)
提供一个基于 Node.js 原生 `http` 模块构建的反向代理 Web Server。
- **核心函数 `createProxyServer(options)`**：创建并返回 HTTP 服务器实例，支持配置上游地址 (`baseUrl`)、模型映射表 (`modelOverride`)、默认路由模型 (`defaultModel`) 以及验证令牌 (`token`)。
- **健康检查端点**：拦截对 `/__cc_proxy_status` 的请求，验证 token 后返回包含 `status: 'ok'` 和进程 PID 的 JSON 数据。
- **请求体处理与模型篡改**：
  - 收集完整的请求体并进行 JSON 解析。
  - 如果请求中包含 `model` 字段，将根据 `modelOverride` 映射表进行替换；若无匹配项且配置了 `defaultModel`，则强制覆盖为默认模型。
  - 重新构建修改后的请求体并计算新的 `content-length` 发往上游。
- **请求转发**：支持 HTTP/HTTPS 上游（对 HTTPS 默认放行自签证书 `rejectUnauthorized: false`），并在上游不可达时返回 `502` 状态码及明确的错误信息。

#### 3. 异步日志写入器 (`logger.js`)
为代理服务器提供高性能、非阻塞的请求日志持久化能力。
- **核心函数 `createLogger(options)`**：返回一个包含 `push`、`flush`、`flushSync`、`stop` 和 `isDisabled` 方法的日志实例。
- **异步队列与 Drain 循环**：
  - `push(entry)`：微秒级同步操作，将日志条目推入内存队列并唤醒异步写入循环。
  - 队列满时采取先进先出（FIFO）淘汰策略，并记录被丢弃的数量。
  - `scheduleDrain()`：使用 `setImmediate` 批量提取队列中的所有条目，合并成多行字符串后执行单次磁盘 I/O，极大降低写盘频率。
- **容错与降级**：可配置连续写入失败的上限 (`maxFailCount`)，达到上限后写入器将自动禁用，避免 I/O 故障拖垮整个代理进程的性能。
- **刷写机制**：
  - `flush(callback)`：异步将剩余日志写入磁盘。
  - `flushSync()`：同步强制写入磁盘，专用于进程崩溃或强制退出 (`SIGTERM`) 等需要立即保存数据的场景。

### 模块依赖关系

- `worker.js` 是整个代理层的根节点，直接依赖于 `server.js` 和 `logger.js`。
- `server.js` 依赖于 Node.js 原生 `http`、`https` 和 `url` 模块。在运行时，它接收由 `worker.js` 注入的 `logger` 实例，在请求结束时调用 `logger.push()` 记录包含请求方法、路径、模型转换情况、状态码和耗时的 JSON 数据。
- `logger.js` 仅依赖 Node.js 原生 `fs` 和 `path` 模块，具备高度独立性，负责将 `server.js` 传入的数据安全地写入本地文件系统。
## API 层

本模块是 CLI 项目的底层 HTTP 通信基础层，主要负责处理与外部 API 端点的网络请求。它封装了 Node.js 原生的 `http` 和 `https` 模块，以 Promise 风格向外提供异步请求能力。

### 模块职责
*   **协议适配**：自动根据传入的 baseURL 判断并适配 HTTP 或 HTTPS 协议。
*   **鉴权处理**：统一处理请求头中的身份验证信息。
*   **数据解析**：自动收集响应数据流，并尝试将响应体解析为 JSON 格式。
*   **网络控制**：提供请求超时中断机制，并对 HTTPS 请求提供自签名证书的支持（关闭严格证书校验）。

### 核心函数

#### `sendApiRequest(baseUrl, token, options)`
向指定的 API 端点发送 HTTP 请求。

**参数说明：**
*   `baseUrl` (string)：API 的基础路径（例如 `'https://api.example.com/v1'`），函数内部会自动去除末尾的斜杠。
*   `token` (string | null)：身份验证令牌。如果传入非空值，将在请求头中添加 `Authorization: Bearer <token>` 和 `x-api-key: <token>`。
*   `options` (Object)：请求配置项。
    *   `path` (string)：请求的具体路径，会与 `baseUrl` 拼接成完整 URL。
    *   `method` (string)：HTTP 请求方法，默认为 `'GET'`。
    *   `timeout` (number)：请求超时时间（毫秒），默认为 `10000`（10秒）。
    *   `body` (string)：请求体数据。如果存在，会自动添加 `Content-Type: application/json` 请求头。

**返回值：**
*   返回一个 `Promise`，resolve 一个包含响应信息的对象：
    *   `statusCode` (number)：HTTP 状态码。
    *   `statusMessage` (string)：HTTP 状态信息。
    *   `durationMs` (number)：请求耗时（毫秒）。
    *   `body` (string)：原始响应体字符串。
    *   `data` (Object | null)：解析后的 JSON 对象。如果响应体不是合法的 JSON 格式，则该字段为 `null`。

**异常处理：**
*   如果 URL 格式不合法，Promise 会 reject 并抛出 `无效的 URL` 错误。
*   如果请求超过设定的 `timeout`，请求连接将被销毁，并 reject 抛出超时错误。
*   网络异常或底层 socket 错误会直接通过 reject 抛出。

### 导出接口
*   `sendApiRequest`：向外暴露的唯一核心请求函数。

### 依赖关系
*   **内部依赖**：该模块通常不独立使用，而是作为底层网络库被项目中的其他业务逻辑层或上层 API 封装模块调用，以实现特定的第三方服务对接。
*   **外部依赖**：重度依赖 Node.js 内置的核心模块 `http` 和 `https` 来实现底层的网络通信，不依赖任何第三方 HTTP 请求库（如 `axios` 等）。
